# daniel-spies.com

Personal site of Daniel Spies. Static HTML / CSS / vanilla JS, hosted on GitHub Pages.

---

## Quick start

```bash
# 1. Serve locally (any static server works; this uses Python's built-in)
npm run serve
# → http://localhost:8000

# 2. Before deploying, regenerate sitemap.xml and robots.txt
npm run build
```

That's it. No bundler, no framework, no runtime dependencies. Node ≥ 18 is the only requirement, and only for the two build scripts under `scripts/`.

## How content works

The single source of truth for articles is **`data/articles.json`**. The home page list, articles index, and sitemap all read from it.

Each entry looks like this:

```json
{
  "slug": "building-personal-cloud",
  "title": "Building a personal cloud at home",
  "subtitle": "How I replaced paid cloud storage with…",
  "date": "2026-05-12",
  "tag": "Project",
  "hero":  "/assets/images/placeholders/hero-cloud.svg",
  "thumb": "/assets/images/placeholders/thumb-cloud.svg"
}
```

Articles are newest-first. The first entry is the "featured" article — the home page lists the next four below it.

`tag` must be exactly one of: `Project`, `Data Analysis`, `Essay`. The articles page filter chips are case-sensitive matches against this field.

### Adding a new article

1. **Copy** `articles/building-personal-cloud.html` to a new file at `articles/<your-slug>.html`.
2. **Edit** the new file:
   - `<title>`, all `<meta>` tags (description, OG, Twitter)
   - `<link rel="canonical">`
   - The BlogPosting JSON-LD block
   - The hero (eyebrow tag, title, subtitle, image)
   - The article meta line (`<time datetime="…">…</time>`)
   - The prose body inside `<div class="prose" data-prose>`
   - The prev/next nav at the bottom
3. **Prepend** a new entry to `data/articles.json` (newest first).
4. **Update `index.html`** if this is now the latest article — swap the hero block (link, image, eyebrow, title, subtitle).
5. **Optionally** add real hero and thumb images under `assets/images/`, and update the JSON paths.
6. **Run** `npm run build` to refresh `sitemap.xml`.
7. **Commit and push**. GitHub Pages picks it up automatically.

### Why is the home featured hero hardcoded?

Because the most prominent content of the page benefits from loading instantly (LCP), indexing cleanly without JavaScript, and not flashing in after a fetch. Updating it costs one extra edit per release; this seemed the right trade.

If you'd rather it be auto-populated from `articles.json`, the change is small — render the first article into `<main>` from `articles.js` before the latest list. Open `assets/js/articles.js` and the comment block in `index.html` for guidance.

## Customisation

### Colours, type, spacing

All design tokens live as CSS custom properties at the top of **`assets/css/base.css`**. Change a colour, type size, or spacing value there and every page picks it up.

The dark theme overrides only colour tokens — structural values (type scale, spacing, layout) stay the same across themes. To shift the dark palette, edit the `[data-theme="dark"]` block in the same file.

### Fonts

The site loads two Google Fonts: **Source Serif 4** (headings) and **Inter** (body / UI). To swap, edit two places:

1. The `<link>` to Google Fonts in every HTML `<head>` (search-and-replace works fine).
2. The two `font-family` stacks in `base.css` (search for `Source Serif 4` and `Inter`).

### Filter categories

If you ever want to add or rename a category:

1. Edit the chips in `articles.html` (`<button class="filter-chip" data-filter="…">`).
2. Use the same value in `data/articles.json` (`tag` field).

The matching is case-sensitive.

### Replacing placeholder images

The `assets/images/placeholders/` SVGs are deterministic gradients generated from each article's slug. Replace them by:

- **For a single article**: drop a real photo at the same path (e.g. `assets/images/placeholders/hero-cloud.svg` → replace with `assets/images/cloud-hero.jpg`, then update the path in `articles.json` and the HTML).
- **For all placeholders**: run `npm run placeholders` to regenerate them (useful if you tweak the gradient palette in `scripts/generate-placeholders.js`).

Recommended hero dimensions: **1600 × 900px**, JPG or WebP, around 150-300 KB after compression. Tools like [Squoosh](https://squoosh.app) handle this in a browser.

## Build scripts

There are two npm scripts. Neither has any dependencies — both use only Node built-ins.

| Command | What it does |
|---|---|
| `npm run build` | Generates `sitemap.xml` and `robots.txt` from `data/articles.json`. Run before each deploy. |
| `npm run placeholders` | Regenerates the placeholder SVGs from `data/articles.json`. Run whenever you change the palette or slug list (rare). |
| `npm run serve` | Starts a local server on port 8000 using Python's `http.server`. |
| `npm run predeploy` | npm convention — runs `build` automatically if a `deploy` script exists. |

No `npm install` is needed. There is no `node_modules/` directory.

## Deployment (GitHub Pages)

1. Create a repository (e.g. `daniel-spies/daniel-spies.github.io` or any name — the `CNAME` handles the custom domain).
2. Push the contents of this folder to the `main` branch.
3. In the repo settings, enable GitHub Pages from `main` / root.
4. In your DNS provider, point `daniel-spies.com` at GitHub Pages:
   - `A` records to `185.199.108.153`, `185.199.109.153`, `185.199.110.153`, `185.199.111.153`
   - `AAAA` records to the IPv6 equivalents (see [GitHub docs](https://docs.github.com/en/pages/configuring-a-custom-domain-for-your-github-pages-site))
   - Or a `CNAME` from `www.daniel-spies.com` to your `<user>.github.io`
5. Enforce HTTPS in repo settings once the certificate provisions.

The `.nojekyll` file prevents GitHub Pages from running its built-in Jekyll processor, which would otherwise hide files/folders starting with `_` or `.`.

## Accessibility

The site targets WCAG 2.1 AA. Things in place:

- Skip-to-content link as the first focusable element on every page
- Semantic HTML: `<header>`, `<nav>`, `<main>`, `<article>`, `<footer>`
- Heading hierarchy: one `<h1>` per page (in the hero), `<h2>` for sections, `<h3>` inside articles
- `aria-current="page"` on the active nav link
- `aria-expanded` and `aria-controls` on the mobile hamburger
- `aria-pressed` on filter chips (not `aria-selected`, which is for tabs)
- `:focus-visible` keyboard-only focus rings
- `prefers-reduced-motion` disables all transitions and animations
- All decorative images have empty `alt=""`; the hero `<img>` is decorative because the headline carries the meaning
- Theme toggle has a dynamic `aria-label` that announces the next state

What's *not* automatic and depends on you:

- **Alt text on real photos** when you replace placeholders. Describe what's *in* the image, not what the article is about.
- **Colour contrast** if you adjust the palette. Use [Stark](https://www.getstark.co) or your browser devtools to verify AA-level contrast.

## Browser support

Modern evergreen browsers — Chrome, Firefox, Safari, Edge. CSS features used that require recent versions:

- CSS custom properties (universal)
- `clamp()`, `min()`, `max()` for fluid type (Safari 11+, all others 2020+)
- `color-mix()` for hover tints (Safari 16.2+, Chrome 111+, Firefox 113+)
- `backdrop-filter` on the nav (Safari 9+, Chrome 76+, Firefox 103+)

If you need to support older browsers, the worst-case fallback is "design looks slightly less polished" — nothing breaks. The cascade and modern reset ensure layout integrity all the way back to ~2018 browsers.

## What's intentionally *not* here

A short list of things this project deliberately avoids:

- **No bundler** (no Webpack, Vite, Parcel). Three CSS files and four JS files don't need bundling.
- **No CSS framework**. Tailwind, Bootstrap, etc. were overkill for ~500 lines of CSS.
- **No JavaScript framework**. The site has four small scripts totalling under 10 KB unminified.
- **No analytics by default**. Add Plausible/Fathom/GA later if needed.
- **No comments engine**. If you ever want comments, GitHub Discussions or an issue-per-article works without changing the build.
